## 1. 项目背景与目标

### 1.1 背景

参考项目 [Remote-sqlite](https://github.com/) 是一款基于 Electron + React 的跨平台远程 SQLite 管理工具，通过 SSH 在远端执行 `sqlite3` 命令来操作数据库。本项目 **RemoteSQLite-Qt** 在此理念基础上，改用 **Qt (QML + C++)** 技术栈重写，并新增 **SFTP 双向同步** 能力——既可以在设备端通过 `sqlite3` 命令直接操作，也可以将数据库文件通过 SFTP 下载到本地操作后上传回设备。

### 1.2 目标

- 使用 Qt 技术栈构建跨平台桌面应用（Windows / Linux / macOS）
- 支持两种远程操作模式：
  - **远程命令模式**：SSH 到设备，调用远端 `sqlite3` 命令（无需传输文件）
  - **本地同步模式**：SFTP 下载 → 本地 sqlite3 操作 → SFTP 上传
- 提供与 [Remote-sqlite] 一致的拟态化（Neumorphism）UI 风格
- 采用严格的分层架构：QML → Model → EventBus → Controller → Driver-Interface → Driver
- 仅 QML 层和 Model 层使用 Qt 框架，其余层使用纯 C/C++

### 1.3 目标用户

| 角色 | 描述 | 主要操作 |
|------|------|----------|
| 嵌入式/IoT 开发者 | 在资源受限设备上使用 SQLite | 远程命令模式（无需传输） |
| 后端/运维工程师 | 远程服务器上的 SQLite 管理 | 两种模式均可 |
| 数据分析师 | 下载到本地进行复杂查询分析 | SFTP 同步模式 |

### 3.2 两种操作模式架构

```
模式 A: 远程命令模式
  QML → Model → EventBus → Controller → ISshDriver (ssh exec)
       → 远端 shell → sqlite3 <db_path> <sql> → 解析 stdout
       → 返回结果

模式 B: SFTP 本地同步模式
  QML → Model → EventBus → Controller → ISftpDriver (sftp get)
       → 下载到本地临时目录
       → ILocalSqliteDriver (本地 sqlite3 操作，直接调用 sqlite3 API)
       → ISftpDriver (sftp put) → 上传回设备
```

## 4. 分层设计

### 4.1 各层职责

| 层 | 语言 | 依赖 | 职责 |
|----|------|------|------|
| **QML 层** | QML + JS | Qt Quick, Qt Quick Controls | UI 渲染、用户交互、动画、主题 |
| **Model 层** | C++ | Qt Core, Qt QML | 数据模型（QAbstractListModel）、属性绑定、信号槽，向上暴露给 QML |
| **EventBus 层** | 纯 C++ | STL, pthread/std::thread | XQQYT/EventBus (git submodule, MIT)，字符串事件名 + 内置线程池 + 异步分发 |
| **Controller 层** | 纯 C++ | EventBus, Driver-Interface | 业务逻辑编排、状态管理、连接池管理、冲突检测 |
| **Driver-Interface 层** | 纯 C | 无 | 抽象接口定义（虚基类 / 函数指针表），只暴露 `.h` |
| **Driver 层** | 纯 C/C++ | libssh2 / OpenSSH / sqlite3 | 具体驱动实现，对接系统库/第三方 C 库 |

### 4.2 依赖方向（自上而下）

```
QML ──依赖──▶ Model ──依赖──▶ EventBus
                                  ▲
Controller ──依赖──▶ EventBus     │
     │                            │
     └──依赖──▶ Driver-Interface ─┘
                     ▲
Driver ──实现──▶ Driver-Interface
```

- **QML 层和 Model 层** 是唯一使用 Qt 的层，它们之间通过 Qt 的信号槽和属性绑定通信
- **Model 层** 通过 EventBus 与下层通信，不直接依赖 Controller
- **Controller 层** 依赖 Driver-Interface（接口），不依赖具体 Driver（依赖倒置）
- **Driver 层** 实现 Driver-Interface，与 Controller 解耦

### 6.7 SFTP 同步模式（本项目的差异化功能）

```
工作流程：
1. 用户选择远程数据库文件
2. 通过 SFTP 下载到本地临时目录
3. 在本地使用原生 sqlite3 C API 操作（更快、更丰富）
4. 操作完成后：
   a. 检查远程文件是否有更新（通过修改时间/文件 hash）
   b. 如有冲突，提示用户选择：覆盖 / 合并 / 放弃
   c. 无冲突则通过 SFTP 上传
5. 清理本地临时文件
```

## 7. 数据流与通信

### 7.1 请求-响应流程（远程命令模式）

```
用户点击"查询表数据"
  → QML 调用 Model.queryTableData(tableName)
  → Model 发送 QueryEvent 到 EventBus
  → DatabaseController 订阅 QueryEvent
  → DatabaseController 构造 sqlite3 命令: "SELECT * FROM {table} LIMIT 100 OFFSET 0"
  → DatabaseController 调用 ISshDriver::execCommand(connectionId, command)
  → LibSsh2Driver 通过 SSH exec channel 执行命令
  → 捕获 stdout，解析 JSON 输出
  → DatabaseController 发送 QueryResultEvent 到 EventBus
  → Model 接收到事件，更新 QAbstractListModel 数据
  → QML 自动刷新表格视图
```

### 7.2 SFTP 同步流程

```
用户点击"下载并编辑"
  → QML 调用 Model.downloadDatabase(dbPath)
  → Model 发送 SftpDownloadEvent
  → SftpController 调用 ISftpDriver::download(remotePath, localPath, progressCallback)
  → 下载完成后发送 SftpDownloadCompleteEvent
  → Model 切换为本地模式
  → 用户进行本地操作（直接走 sqlite3 C API，更快）
  → 用户点击"同步到设备"
  → SyncController 检查冲突 → 调用 ISftpDriver::upload(localPath, remotePath)
  → 上传完成后发送 SyncCompleteEvent
```

### 7.3 EventBus 设计要点（基于 XQQYT/EventBus 子仓库）

子仓库路径：`libs/EventBus`，MIT 协议

**核心 API**：
- `initEventBus(EventBusConfig{ThreadModel, TaskModel, thread_min, thread_max, task_max})` — 初始化，内置线程池
- `registerEvent("EventName")` — 注册字符串事件名
- `subscribe("EventName", [](auto... args){...})` — 订阅事件（自动推导类型），返回 `callback_id`
- `subscribeSafe("EventName", callback)` — 自动注册+订阅（事件未注册时不会抛异常）
- `publish("EventName", args...)` — 发布事件，**全异步**，通过线程池分发
- `publishWithPriority(priority, "EventName", args...)` — 优先级发布（需 PRIORITY 任务模型）
- `unsubscribe("EventName", callback_id)` — 取消订阅

**配置项**：
- `ThreadModel::FIXED`：固定线程数（适合资源受限场景）
- `ThreadModel::DYNAMIC`：弹性伸缩线程数（默认推荐）
- `TaskModel::NORMAL`：FIFO 任务队列
- `TaskModel::PRIORITY`：优先级队列（HIGH/MIDDLE/LOW）

**使用方式**：字符串事件名标识事件，`std::function<void(Args...)>` 作为回调，`std::any` 做类型擦除存储。所有 `publish()` 调用在线程池中异步执行回调。

## 9. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| LibSSH2 API 复杂度高 | SSH/SFTP 开发周期长 | 封装清晰的 C 接口，提供 Mock Driver 并行开发上层 |
| QML 拟态化阴影性能 | 低配机器渲染卡顿 | 使用 `layer.enabled: true` 缓存、限制阴影层级 |
| SFTP 大文件传输超时 | 用户体验差 | 分块传输 + 断点续传 + 进度回调 |
| 远端 sqlite3 版本不一致 | 命令输出格式差异 | 启动时检测版本，适配不同 JSON 输出格式 |
| Windows 路径兼容 | SFTP 路径解析错误 | 统一使用 POSIX 风格路径，由 Driver 层处理转换 |
| Qt 与纯 C++ 层边界管理 | 内存泄漏/双重释放 | 严格的 RAII、智能指针在 C++ 侧，C 接口使用明确的所有权语义 |
